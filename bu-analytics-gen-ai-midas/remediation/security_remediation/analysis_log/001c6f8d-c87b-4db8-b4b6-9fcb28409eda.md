## Root cause
Dockerfile Misconfiguration: Default User Privilege in unknown Dockerfile (path truncated by scanner). The Dockerfile does not include a USER instruction so the container process runs as root (uid 0). Running as root means that if container isolation is breached, the attacker has root on the host. The frontend/Dockerfile already uses USER node — the backend and ECS-specific Dockerfiles need the same treatment.

## Remediation plan
1) Add a non-root user in each affected Dockerfile:
   RUN addgroup -S appgroup && adduser -S appuser -G appgroup
   USER appuser
2) Ensure WORKDIR and runtime files are chown'd to appuser before USER instruction.
3) Verify application does not require root at runtime (ports 8000/8080 are non-privileged).
4) Add Checkov CKV_DOCKER_2 (non-root user) to CI.
