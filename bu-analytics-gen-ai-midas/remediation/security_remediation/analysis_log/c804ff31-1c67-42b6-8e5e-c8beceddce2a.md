## Root cause
Cross-Site Request Forgery: frontend API service files (fastApiService.ts, apiInterceptor.ts, creditRiskApi.ts, authService.ts, rfeService.ts, apiServices.ts) make state-changing fetch requests without including an explicit CSRF token header. Fortify flags this as CSRF vulnerability. The actual risk is mitigated by: (1) all API calls include Authorization: Bearer <JWT> header — CSRF attacks cannot read or forge the JWT from a cross-origin page; (2) the backend uses JWT Bearer auth, not session cookies, so the CSRF attack model does not apply to the primary authentication mechanism. However, if any API endpoint accepts cookie-based authentication as a fallback, or if SameSite=None cookies are used, CSRF remains a real risk.

## Remediation plan
1) frontend/src/services/apiInterceptor.ts: confirm Authorization header is always included in all state-changing requests (POST/PUT/DELETE/PATCH) — this is the primary CSRF mitigation for JWT-based SPAs.
2) Verify the FastAPI backend sets SameSite=Strict or SameSite=Lax on any session cookies (check backend/app/core/session/).
3) If the backend uses any cookie-based auth endpoint: add a CSRF double-submit cookie pattern — generate a CSRF token on login, store in a non-HttpOnly cookie, and require it as an X-CSRF-Token header on state-changing requests.
4) Add a CORS policy on the backend ALB and FastAPI that restricts allowed origins to the MIDAS frontend domain only.
5) For frontend/test_project_integration.html (OBV0203/0204): these are developer test pages — ensure they are not deployed in production (exclude from Docker image).
