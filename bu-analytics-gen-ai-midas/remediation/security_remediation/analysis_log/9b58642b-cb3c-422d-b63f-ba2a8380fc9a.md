## Root cause
docs/html_diagram/ contains multiple static HTML architecture diagrams that assign data-attribute values and URL parameters directly to innerHTML. Specific instances: (1) future/developer/index.html lines 942, 964, 1000-1001, 1011-1016: tt.innerHTML = h where h is built from data-* attributes (el.dataset.*) via template literals without HTML encoding; dpDesc.innerHTML=fmt(...) where fmt() uses escapeHtml but then re-injects <br/> via replace. (2) future/requirements/index.html line 129: filterText.innerHTML built from window.location.search param 'node' via esc() - safe only if esc() encodes all special chars. (3) future/architecture.html line 812: tooltip.innerHTML built from el.dataset values. The risk: if any data-* attribute or URL parameter is attacker-controlled, XSS executes. These are internal documentation pages but served over the same origin.

## Remediation plan
1) Audit each fmt() and esc() helper in the html_diagram files to confirm they encode <, >, &, ', and " - add a unit test if feasible.
2) For all tooltip innerHTML assignments (tt.innerHTML = h), validate every variable in the h template literal passes through escapeHtml/esc before insertion.
3) docs/html_diagram files should not be served at the application origin - move behind an authenticated internal path or serve as a separate static site with restrictive Content-Security-Policy.
4) Add sandbox attribute to any iframe embedding these diagrams.
5) Evaluate whether these docs pages need to be deployed to prod at all; if not, exclude from the ECR image.
