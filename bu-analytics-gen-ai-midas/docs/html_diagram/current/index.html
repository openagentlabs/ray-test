Current Architecture (Simplified)

A simplified developer view of the current backend. This page intentionally avoids deep detail — it exists to clarify what is running today and why it must be refactored into the future-state microservice architecture.

🧱 Current architecture diagram

Client

Web UI and users call a single backend API.

WORKS

Ingress

Direct HTTP traffic to the monolith (limited gateway/mesh policy).

LIMITED CONTROLS

FastAPI Monolith

One codebase owns everything: upload, datasets, QC/DQS, chat, training, GraphRAG, documentation, evaluation. Routing concentrated in routes.py.

SCALING BOTTLENECK

Process memory state

In-memory DataFrames and session selections live inside the pod/process.

NOT HORIZONTALLY SCALABLE

Background threads

Long-running ML work is executed via threads without durable job state or retries.

FRAGILE

State stores (mixed)

SQLite/Postgres fallback, some S3/local filesystem usage, partial Redis usage.

INCONSISTENT

Vector store on disk

FAISS index stored on local disk → replicas diverge and restart loses index.

DURABILITY RISK

⚠️ Current pain points (developer impact)

In-memory DataFrames
  • Singleton state prevents safe horizontal scaling.
  • Restarts lose “current dataset state”.
In-memory session LLM selection
  • Lost on pod restart.
  • Forces sticky sessions to keep behavior stable.
BackgroundJobManager threads
  • No distributed visibility.
  • No robust retry semantics.
  • Long-running ML blocks request threads.
FAISS on local disk
  • Not shared across replicas.
  • Index diverges and is lost on restart.

➡️ Why the future state is microservices

  • Stateless compute + managed state: move all durable state to DynamoDB/Redis/S3/S3 Files.
  • Clear ownership: one service owns each data domain; other services use APIs.
  • Robust ML execution: move training to Kubeflow Pipelines (step pods with retry/logging).
  • Security and governance: Istio mTLS, identity/authz split, WAF+ALB ingress, Secrets Manager.
  • Cloud portability: ports/adapters pattern (no direct AWS SDK imports in domain code).
