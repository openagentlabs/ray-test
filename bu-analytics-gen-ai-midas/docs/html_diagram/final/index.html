EXLDecision — Future State Architecture

Decomposition of the FastAPI monolith into 14 independent microservices on AWS EKS + Istio, with an AI Agent Fabric built on AWS Bedrock AgentCore. This is the software developer view — every service, file, endpoint, and data handoff.

Status: Proposed Audience: Software Developers Platform: AWS EKS us-east-1 Services: 14 microservices Date: 2026-04-28

📐 Design Principles

1 — Stateless Pods
No pod holds durable state. A pod can die, be rescheduled, or replicated with zero data loss.
2 — One Service, One DB Namespace
Services never share DynamoDB tables or Redis keyspaces directly — always via the owning service's API.
3 — Event-Driven for Long Work
Any operation >3 s returns a job ID immediately. Kubeflow / SQS handles async execution.
4 — Cloud-Portable Abstractions
Code imports StoragePort, CachePort, NoSQLPort — never boto3 directly. Swap AWS → Azure/GCP by changing the adapter.
5 — Istio Owns Traffic Policy
mTLS between all services. Retry/circuit-breaker in VirtualService/DestinationRule — no per-service logic.
6 — Secrets Manager Only
No env vars for secrets. Pods receive a secret reference; abstraction layer fetches at startup.
7 — DynamoDB First + Redis Cache
Postgres/SQLite replaced by DynamoDB. Hot reads cached in ElastiCache Redis with TTL.
8 — ML Artefacts on S3 Files
No binary blobs in DynamoDB. Data layer writes parquet/pickle/models to S3 Files; metadata to DynamoDB.
9 — gRPC Inside / REST Outside
Service-to-service: gRPC (HTTP/2, Protobuf). External clients (browser, CLI): RESTful HTTP/1.1 JSON.
10 — AI Agents as First-Class Citizens
Every EXLDecision capability is a tool callable by an AI agent via MCP. Agents use the same REST/gRPC APIs as humans.

gRPC (Protobuf, HTTP/2)

🗄️ Storage Strategy

DynamoDB
Use for: All structured metadata, user records, RBAC, pipeline catalogue, run state, evaluation metadata.

Never store: Binary blobs, DataFrames, pickled objects.
ElastiCache Redis
Use for: Session tokens, permission cache (5min), DataFrame hot cache (30min), LLM selection, progress pub/sub.

Policy: allkeys-lru, 3-node cluster, keyspace notifications on.
S3 Files
Use for: Parquet datasets, splits, model artefacts, evaluation files, knowledge graphs, agent run outputs.

Access: Only via data-fabric-service gRPC — never direct boto3 in other services.
AgentCore Memory
Use for: Long-term agent memory — user preferences, dataset names, model choices across sessions.

Access: Only via AgentRuntimePort.store_memory / retrieve_memory — never DynamoDB direct.
Secrets Manager
Use for: All credentials, API keys, JWT signing keys, AgentCore credentials.

Rule: Never in env vars, never in code, never in images.
OpenSearch Serverless
Use for: Vector index for semantic search and RAG context retrieval (replaces FAISS on local disk).

Owned by: graphrag-service only.
